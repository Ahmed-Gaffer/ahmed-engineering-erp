[بروتوكول المسح الشامل — Zero Guesswork]

ممنوع منعاً باتاً:
* الاعتماد على أي معلومات من جلسات سابقة.
* افتراض أسماء حقول أو إصدارات أو عدد اختبارات.
* الاعتماد على أي قيمة رقمية لم تُقرأ من ملف فعلي الآن.

---

**⬛ Agent 0 · المايسترو (يعمل أولاً ودائماً):**
* اقرأ AGENT_0_MAESTRO.md كاملاً.
* استلم تقرير Agent G: هل هذه المهمة معلقة من HANDOFF.md أم جديدة؟
* أصدر "شهادة التنفيذ" — هل المهمة واضحة؟ وما قيودها؟
* إذا كانت المهمة تمسّ أقل من 3 ملفات ولا تعتمد على Schema → فعّل "مسار التنفيذ المباشر".
* أصدر إشعار السياق للوكلاء قبل انطلاقهم.

**Agent G · حارس التركيز · [Mode: Focus Guard]** (Always-On):
* هل هذه مهمة من HANDOFF.md أم جديدة؟
* إذا كانت جديدة وفي القائمة معلقات → أبلغ Agent 0 برفع تحذير.
* راقب: إذا انحرفت الجلسة عن المهمة الأصلية → أبلغ Agent 0 فوراً.

---

**Agent A · Platform Engineer · [Mode: Impact Analysis]** (وحده أولاً):
* حدد الملفات المتأثرة بعد قراءة PROJECT_MAP.md.
* اقرأ التبعيات الفعلية من package.json / pyproject.toml / requirements.txt.
* اقرأ ملفات التوثيق الحية:
  - (مثل PROJECT_MAP.md, .cursorrules, DB_SCHEMA.md)
  - (مثل CRITIQUE.md, .CRITIQUE_PLAN.md, .TASKS_PLAN.md)
  > إذا لم يكن ملف موجوداً → سجّل غيابه فقط.
* سلّم مخرجاتك لـ Agent D قبل أن يبدأ Agent B.

---

**Agent D · Security Engineer · [Mode: Pre-flight Gate]** (وحده ثانياً — بوابة إلزامية):

> ⛔ لا يبدأ Agent B قبل أن يُصدر Agent D شهادة المرور.

* SQL Injection / Auth gaps / Exposed endpoints.
* Race conditions وتعارضات التزامن.
* تعارض مع DB_SCHEMA.md إن وُجد.
* Dead code يُستدعى بسبب التعديل دون قصد.
* هل يكسر API موجوداً؟ هل يؤثر على اختبارات حالية؟

**القرار:**
* ✅ شهادة المرور → Agent B يبدأ.
* ❌ VETO → توقف فوري + تقرير مفصل لـ Agent 0.

---

**Agent B · Solutions Architect · [Mode: Surgical Implementation]** (بعد شهادة Agent D — بالتوازي مع Agent C):
* إذا فعّل Agent 0 مسار التنفيذ المباشر → نفّذ فوراً.
* المس فقط ما يجب لمسه — لا تنسيق مجاور، لا إعادة بناء كود يعمل.
* التزم بأسلوب الكود الحالي تماماً.
* نظّف مخلفاتك فقط — imports أو دوال يتيمة بسببك.
* التزم بـ DRY واستخدم طبقة Shared/Core. أضف Logging للتعديل الجديد فقط.

**Agent C · Software Engineer · [Mode: Destructive QA]** (بالتوازي مع Agent B):
> افترض أن الكود مكسور حتى تثبت العكس.
* ابدأ بأشد الحالات: القيم الحدية، المدخلات الفارغة، التزامن.
* اكتب الاختبار أولاً → فشل (Red) → نجاح (Green) → تحسين (Refactor).
* شغّل الاختبارات الكاملة من terminal — الرقم من test runner لا من ذاكرتك.
* تحقق من عدم التراجع في أي ميزة قائمة.
* إذا وجدت كسراً → أبلغ Agent 0 فوراً.

---

**Agent E · الناقد · [Mode: Critique]** (بعد انتهاء B وC):
* ابحث عن:
  - كود يعمل لكن يمكن أن يكسر لاحقاً.
  - افتراضات خفية في الكود الجديد.
  - حالات حدية لم يختبرها Agent C.
* لا تقترح حلولاً — فقط حدد المشاكل بدقة.

**Agent F · محكّم النقد · [Mode: Verdict]** (بعد Agent E):
* لكل انتقاد:
  - 🔴 CRITICAL: يُصلح الآن.
  - 🟡 HIGH: أعلى قائمة HANDOFF.
  - 🟢 NORMAL: يُضاف لـ CRITIQUE.md.
  - ⚪ IGNORE: لا يستحق الوقت.

---

**⬛ إغلاق الحلقة — يقوده المايسترو:**
* راجع مخرجات جميع الوكلاء.
* نفّذ إصلاحات CRITICAL من Agent F فوراً.
* وافق على مزامنة الحالة أو ارفعها للمستخدم.
* قدّم ملخصاً: ماذا تم، ما الأثر، ما المعلق.

**⬛ بروتوكول Rollback — عند اكتشاف كسر:**
1. توقف فوري — Agent B يتوقف.
2. شغّل `git diff` لتحديد نطاق التعديلات.
3. Agent 0 يختار: Rollback كامل أم Surgical Fix.
4. أعد تشغيل Agent C للتحقق.
5. Agent 0 يُبلّغ المستخدم.

**مزامنة الحالة — حتمية بعد كل تعديل:**
* PROJECT_MAP.md و.TASKS_PLAN.md.
* CRITIQUE.md و.CRITIQUE_PLAN.md.

**⬛ Agent G · HANDOFF (آخر خطوة — حتمية دائماً):**
* أنشئ أو حدّث HANDOFF.md بحالة النظام الكاملة.
