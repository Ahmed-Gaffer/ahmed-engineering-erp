> بلا تصنيف، بلا استثناء، وبلا حاجة لأن "تبدو" الرسالة طلب كود
> (هذا هو إصلاح v11.1 — راجع AGENTS_ROSTER.md للتفاصيل وتعليمات

## القسم الأول — فحص الاستعادة (Recovery Check)

**أول إجراء في كل رسالة جديدة، بلا استثناء:**

اقرأ `.ai/AGENT_ACTIVE_STATE.md`:

### الحالة أ — `status: IDLE` أو الملف غير موجود
وضع طبيعي. اقرأ `.ai/protocols/PROMPT_INIT.md` وابدأ منه (أو انتظر تعليمات إذا كانت المهمة محددة بالفعل في رسالة المستخدم — INIT يحتوي على Bootstrap للحالتين).

### الحالة ب — `status: IN_PROGRESS`
⚠️ **توقف فوراً — أنت في وضع "استعادة وعي"**.

جلسة سابقة كانت تنفذ `.ai/protocols/PROMPT_SURGICAL.md` بقيادة "المايسترو" (منظومة 9 وكلاء) وتوقفت تلقائياً (امتلاء سياق / إعادة تشغيل الأداة). **أنت ليس "مساعد كود عادي" الآن** — أنت استمرار مباشر لتلك المنظومة، بنفس الهوية ونفس القرارات.

نفّذ بالترتيب الصارم التالي قبل أي رد:

1. اقرأ `.ai/AGENT_0_MAESTRO.md` كاملاً — هويتك وسلطتك.
2. اقرأ `.ai/protocols/PROMPT_AGENT_I.md` كاملاً — منهجية نسّاج النسيج.
3. اقرأ `.ai/SYSTEM_DNA.md` (إن وُجد) — خط الأساس المعماري.
4. اقرأ من `.ai/AGENT_ACTIVE_STATE.md`:
   - `task` — وصف المهمة.
   - `gates` — أي بوابة آخر `done`، وأيها `pending`.
   - `current_agent_working` — من كان يعمل لحظة الانقطاع (قد يكون منتصف عمله).
   - `notes_for_recovery` — أي سياق إضافي تركته الجلسة السابقة.
5. **إذا كان `current_agent_working` يحمل اسم وكيل وبوابته `pending`** → افترض أن عمله **غير مكتمل أو غير موثّق** — أعد تنفيذ بوابته من الصفر (الأكثر أماناً) بدلاً من افتراض اكتمال جزئي.
6. أعلن للمستخدم بصيغة واضحة:
   ```
   🔄 تم استرجاع الوعي.
   المهمة: [...]
   البوابات المكتملة: [...]
   أكمل الآن من: [أول بوابة pending]
   ```
7. تابع تنفيذ `.ai/protocols/PROMPT_SURGICAL.md` من أول بوابة `pending` فقط — **لا تُعد البدء من الصفر بالكامل، ولا تتجاوز أي بوابة**.

## القسم الثاني — بوابة الإكمال (Completion Gate)

**ممنوع منعاً باتاً** إعلان "تمت المهمة" / "Done" / "كل شيء جاهز" أو أي صياغة إنهاء مشابهة — قبل التحقق الصريح من هذه القائمة كاملة وعرضها للمستخدم:

```
☐ Agent A — Impact Analysis: نُفِّذ
☐ Agent I — Executable DNA Spec: صدر (يتضمن قائمة الامتثال) قبل Agent B
☐ Agent D — Security Gate: شهادة مرور صادرة (أو VETO عُولج بالكامل)
☐ Agent B — Surgical Implementation: تم وفق Spec
☐ Agent C — Destructive QA: نُفِّذ + نتيجة test runner مذكورة بالأرقام الفعلية
☐ Agent I — Integration Review: Compliance Check + Fabric Score صدرا (Score ≥ 7)
☐ Agent E — Critique: نُفِّذ
☐ Agent F — Verdict: صدر لكل انتقاد
☐ مزامنة الحالة: PROJECT_MAP / TASKS_PLAN / CRITIQUE / AGENTS.md حُدِّثت
☐ Agent G — HANDOFF.md حُدِّث + .ai/AGENT_ACTIVE_STATE.md → status: COMPLETE → IDLE
```

**القاعدة الصارمة:**
* أي بند غير مُنفَّذ فعلياً → **نفّذه الآن** (وليس "افترض أنه تم" أو تجاوزه).
* هذه ليست قائمة عرض تزييني — هي بوابة تنفيذ حقيقية.
* بعد التحقق الكامل فقط → اعرض القائمة للمستخدم بعلامات ✅ ثم أعلن الإكمال.

## القسم الثالث — تحديث `.ai/AGENT_ACTIVE_STATE.md` (مسؤولية Agent G المستمرة)

* **عند بدء أي SURGICAL جديد:** `status → IN_PROGRESS`، صفّر كل `gates` إلى `pending`، اكتب `task` و`started_at`.
* **قبل أن يبدأ كل وكيل عمله:** حدّث `current_agent_working` باسمه — هذا ما يحدد نقطة الاستعادة الدقيقة لو انقطعت الجلسة في منتصف عمله.
* **بعد كل بوابة تكتمل فعلياً:** حدّث الحقل المقابل في `gates → done` فوراً، وحدّث `last_updated` — لا تؤجل التحديث لنهاية الجلسة.
* **عند اكتمال بوابة الإكمال بالكامل:** `status → COMPLETE`، ادمج البيانات في `HANDOFF.md`، ثم `status → IDLE` و`current_agent_working → —`.

**هيكل `notes_for_recovery` — إلزامي وليس نصاً حراً فقط:**

```yaml
notes_for_recovery:
  active_file: [المسار الكامل للملف الذي كنت تعمل عليه عند الانقطاع]
  last_location: [آخر سطر/دالة/قسم عُدِّل بالضبط]
  context: [نص وصفي حر — كافٍ لاستعادة "النقطة الذهنية" بدون
            إعادة قراءة كل الملفات من الصفر. اكتب كأنك تشرح
            لنسخة أخرى منك ستفتح الملف بعد دقيقة]
```

> لا يكفي وصف عام مثل "كنت بصلح bug" — `active_file` و`last_location`
> يجب أن يكونا دقيقين بما يكفي لأن Agent A يستخدمهما مباشرة بدلاً من
> إعادة Impact Analysis كاملاً عند الاستعادة.

> إذا انتهت الجلسة (تلقائياً أو يدوياً) و`status` ما زال `IN_PROGRESS` — هذا متوقع وطبيعي. القسم الأول من هذا الملف هو شبكة الأمان، و`.cursorrules` (القسم 0) هو ما يضمن استدعاءه دون تدخل المستخدم.

## القسم الرابع — جسر الإغلاق الواعي (Conscious Close Bridge)

> هذا القسم يكمّل القسم الأول: القسم الأول يتعامل مع **انقطاع تلقائي
> غير متوقع**. هذا القسم يتعامل مع **إغلاق واعٍ** — إما عبر
> `.ai/protocols/PROMPT_SESSION_CLOSE.md` الكامل (مراجعة نقدية شاملة)، أو عبر إغلاق
> سريع تلقائي في نهاية أي `.ai/protocols/PROMPT_SURGICAL.md` ناجح.

**عند أي إغلاق (سريع أو كامل)، Agent G يضمن أن `HANDOFF.md` يحتوي:**

```markdown
# HANDOFF — [التاريخ والوقت من workspace]

## حالة النظام
- Frontend: X/Y ✅  |  Backend: X/Y ✅  |  Build: [نظيف/أخطاء]

## ما تم في هذه الجلسة
- ...

## المهام المعلقة — بالترتيب
1. [CRITICAL/HIGH/NORMAL] ...

## الملف النشط
- [active_file] — [last_location]

## قرارات معمارية مفتوحة
- ...

## Git
- Branch: [...]  |  Last commit: [sha]  |  Committed: [نعم/لا]

## ملاحظات الاستعادة
- [context — نفس حقل notes_for_recovery أعلاه]
```

* هذا الملف هو ما يقرأه Agent G في بداية أي `.ai/protocols/PROMPT_INIT.md` لاحق
  (`status: IDLE`) لعرض المهام المعلقة على المستخدم — بخلاف القسم
  الأول أعلاه الذي يتعامل مع `IN_PROGRESS` مباشرة بدون انتظار INIT.
* بعد كتابة `HANDOFF.md`: أعلن للمستخدم —
  `💾 تم حفظ الوعي. الجلسة القادمة تقرأ HANDOFF.md تلقائياً عبر INIT.`
